# SPEC: Content Negotiation — Multi-Format Response from Same Endpoints

> **Status: Architecture Finalized (2026-04-03)** — Revised after design review. Original research validated. PageRenderer + template architecture replaces ContentTransform approach. Implementation NOT started.

---

## Guiding Principle

> **Build-time code generation over runtime framework features.** Where behavior can be determined at build time and the security surface is small enough to audit and fuzz-test exhaustively, generate the code directly rather than depending on framework libraries. Reserve framework dependencies for infrastructure concerns (HTTP serving, TLS, connection pooling) that are genuinely hard to implement correctly and where the security surface is too large to own.

**Rationale**: Framework templating engines (Thymeleaf, Jinja2, ERB) introduce version coupling, CVE exposure (SSTI attacks), and lock each generated server to a framework-specific rendering model. A generated PageRenderer with centralized `escapeHtml()` is auditable in seconds, fuzz-testable by LLMs, and portable across all target languages without upstream dependencies.

---

## Problem

The generated REST API returns JSON only. Different clients need different formats from the same endpoints:

- `Accept: application/json` → JSON (current behavior, no change)
- `Accept: text/html` → Server-rendered HTML pages with live data (tables, forms, detail views)
- `Accept: application/xml` → XML (future — same PageRenderer pattern as HTML)
- `Accept: text/csv` → CSV (future — same PageRenderer pattern as HTML)

Today, `generated-html/` produces static HTML scaffolds with no live data. Content negotiation makes those pages dynamic — same structure, real data, served from the same REST endpoints that already serve JSON.

**MVP focus: JSON + HTML only.** XML and CSV follow the same pattern once HTML is proven — they are not in MVP scope.

---

## Core Architectural Decision

**The HTTP framework handles content negotiation (Accept header parsing, format routing). appget generates PageRenderer classes and static template files for content transformation.**

| Responsibility | Owner | What It Does |
|---------------|-------|-------------|
| **Content Negotiation** | HTTP framework (Spring Boot, FastAPI, Gin, Sinatra, Rocket) | Reads `Accept` header, selects response format, sets `Content-Type`, routes to correct handler method |
| **Page Structure** | appget-generated template files (pipeline-level artifact) | Static HTML layouts with minimal `{{CONTENT}}` placeholders — no iteration, no escaping, no logic |
| **Data Rendering** | appget-generated `*PageRenderer` classes (per language) | Builds dynamic HTML content (table rows, form inputs) with proper escaping. Loads template, substitutes `{{CONTENT}}`, returns final HTML string |
| **Security (XSS)** | Centralized `escapeHtml()` utility (per language) | Single source of truth for HTML escaping. Auditable, fuzz-testable, no framework dependency |

**Why this split?**
- Framework handles Accept header parsing — reimplementing it is wasted effort and a source of bugs
- Template files are dumb static layouts — pipeline-level artifacts alongside `.proto` and `openapi.yaml`
- PageRenderers are generated at build time with full model knowledge — no runtime type detection needed
- `escapeHtml()` isolates the security boundary in one auditable location per language
- No dependency on framework templating engines (Thymeleaf, Jinja2, ERB) — eliminates version coupling and CVE surface

## Architecture: Two Artifacts

### Artifact 1: Template Files (pipeline-level, language-agnostic)

Static HTML files with a single `{{CONTENT}}` placeholder. No iteration, no conditionals, no escaping — the template is deliberately "dumb."

Generated by `HtmlCrudGenerator` (already exists for Phase 0d static pages). The same generator is extended to emit templates with `{{CONTENT}}` placeholders where dynamic data belongs.

```
Pipeline artifacts (language-agnostic):
├── models.yaml
├── specs.yaml
├── openapi.yaml
├── *.proto
└── templates/                    ← NEW
    ├── index.html                ← root nav (all domains + entities)
    └── {domain}/{table}/
        ├── list.html             ← table structure + {{CONTENT}} for rows
        ├── detail.html           ← detail layout + {{CONTENT}} for field values
        ├── edit.html             ← form layout + {{CONTENT}} for pre-filled inputs
        └── create.html           ← form layout (fully static — no placeholder needed)
```

**Example — `templates/social/users/list.html`:**

```html
<!DOCTYPE html>
<html>
<head><title>Users</title></head>
<body>
  <nav><a href="/">Home</a> &gt; <a href="/api/social">Social</a></nav>
  <h1>Users</h1>
  <a href="/api/social/users?action=create">Create New</a>
  <table>
    <thead>
      <tr><th>ID</th><th>Email</th><th>Username</th><th>Actions</th></tr>
    </thead>
    <tbody>
      {{CONTENT}}
    </tbody>
  </table>
</body>
</html>
```

**Example — `templates/social/users/create.html`:**

```html
<!DOCTYPE html>
<html>
<head><title>Create User</title></head>
<body>
  <nav><a href="/">Home</a> &gt; <a href="/api/social/users">Users</a></nav>
  <h1>Create User</h1>
  <form method="POST" action="/api/social/users">
    <label for="email">Email</label>
    <input type="text" name="email" id="email" required>
    <label for="username">Username</label>
    <input type="text" name="username" id="username" required>
    <button type="submit">Create</button>
  </form>
  <a href="/api/social/users">Cancel</a>
</body>
</html>
```

The create form is fully static. Zero `{{CONTENT}}` — no data substitution needed at runtime.

### Artifact 2: PageRenderer Classes (per language, per model)

Generated at build time by `ServerEmitter`. Each model gets a `*PageRenderer` class that:

1. Loads its template files (once, at startup)
2. Has methods per page type: `renderList()`, `renderDetail()`, `renderEditForm()`
3. Builds the dynamic `{{CONTENT}}` portion using StringBuilder with proper escaping
4. Returns the final HTML string via `template.replace("{{CONTENT}}", renderedContent)`

**Example — `UsersPageRenderer.java` (generated):**

```java
public class UsersPageRenderer {
    private final String listTemplate;
    private final String detailTemplate;
    private final String editTemplate;
    private final String createTemplate;

    public UsersPageRenderer() {
        this.listTemplate = loadTemplate("templates/social/users/list.html");
        this.detailTemplate = loadTemplate("templates/social/users/detail.html");
        this.editTemplate = loadTemplate("templates/social/users/edit.html");
        this.createTemplate = loadTemplate("templates/social/users/create.html");
    }

    public String renderList(List<Users> users) {
        StringBuilder rows = new StringBuilder();
        for (Users u : users) {
            rows.append("<tr>")
                .append("<td>").append(escapeHtml(String.valueOf(u.getId()))).append("</td>")
                .append("<td>").append(escapeHtml(u.getEmail())).append("</td>")
                .append("<td>").append(escapeHtml(u.getUsername())).append("</td>")
                .append("<td><a href=\"/api/social/users/")
                    .append(u.getId()).append("\">View</a></td>")
                .append("</tr>\n");
        }
        return listTemplate.replace("{{CONTENT}}", rows.toString());
    }

    public String renderDetail(Users user) {
        StringBuilder content = new StringBuilder();
        content.append("<dl>")
            .append("<dt>ID</dt><dd>").append(escapeHtml(String.valueOf(user.getId()))).append("</dd>")
            .append("<dt>Email</dt><dd>").append(escapeHtml(user.getEmail())).append("</dd>")
            .append("<dt>Username</dt><dd>").append(escapeHtml(user.getUsername())).append("</dd>")
            .append("</dl>")
            .append("<a href=\"/api/social/users/").append(user.getId())
                .append("?action=edit\">Edit</a>");
        return detailTemplate.replace("{{CONTENT}}", content.toString());
    }

    public String renderEditForm(Users user) {
        StringBuilder content = new StringBuilder();
        content.append("<input type=\"hidden\" name=\"_method\" value=\"PUT\">")
            .append("<input type=\"hidden\" name=\"id\" value=\"")
                .append(user.getId()).append("\">")
            .append("<label for=\"email\">Email</label>")
            .append("<input type=\"text\" name=\"email\" id=\"email\" required value=\"")
                .append(escapeHtml(user.getEmail())).append("\">")
            .append("<label for=\"username\">Username</label>")
            .append("<input type=\"text\" name=\"username\" id=\"username\" required value=\"")
                .append(escapeHtml(user.getUsername())).append("\">")
            .append("<button type=\"submit\">Save</button>");
        return editTemplate.replace("{{CONTENT}}", content.toString());
    }

    public String renderCreateForm() {
        return createTemplate;  // fully static — no substitution needed
    }

    private static String escapeHtml(String value) {
        return HtmlEscapeUtils.escape(value);
    }

    private static String loadTemplate(String path) {
        // Load from classpath resources at startup
        return new String(getClass().getClassLoader()
            .getResourceAsStream(path).readAllBytes(), StandardCharsets.UTF_8);
    }
}
```

### Security Utility: `HtmlEscapeUtils` (one per language)

Centralized HTML escaping. Single source of truth for XSS prevention across all PageRenderers.

```java
public final class HtmlEscapeUtils {
    private HtmlEscapeUtils() {}

    public static String escape(String value) {
        if (value == null) return "";
        StringBuilder sb = new StringBuilder(value.length());
        for (int i = 0; i < value.length(); i++) {
            char c = value.charAt(i);
            switch (c) {
                case '&':  sb.append("&amp;");  break;
                case '<':  sb.append("&lt;");   break;
                case '>':  sb.append("&gt;");   break;
                case '"':  sb.append("&quot;"); break;
                case '\'': sb.append("&#x27;"); break;
                default:   sb.append(c);
            }
        }
        return sb.toString();
    }
}
```

**Per-language implementations** follow the same 5-character replacement logic. Peel back from battle-tested framework implementations (Spring's `HtmlUtils.htmlEscape()`, Python's `html.escape()`, Go's `html.EscapeString()`, Ruby's `ERB::Util.html_escape()`) to ensure correctness.

## Controller Changes

Controllers gain `produces` annotations to declare supported formats. The framework handles Accept header matching and routes to the correct handler method.

**Pattern**: Separate methods per format. The JSON method returns data as-is (Jackson/protobuf handles serialization). The HTML method calls the PageRenderer.

```java
// Generated: UsersController.java

@GetMapping(produces = MediaType.APPLICATION_JSON_VALUE)
public ResponseEntity<List<Users>> listJson() {
    return ResponseEntity.ok(repository.findAll());
}

@GetMapping(produces = MediaType.TEXT_HTML_VALUE)
public ResponseEntity<String> listHtml() {
    String html = usersPageRenderer.renderList(repository.findAll());
    return ResponseEntity.ok(html);
}

@GetMapping(value = "/{id}", produces = MediaType.APPLICATION_JSON_VALUE)
public ResponseEntity<Users> getJson(@PathVariable int id) {
    Users user = repository.findById(id);
    return ResponseEntity.ok(user);
}

@GetMapping(value = "/{id}", produces = MediaType.TEXT_HTML_VALUE)
public ResponseEntity<String> getHtml(@PathVariable int id,
                                       @RequestParam(defaultValue = "view") String action) {
    Users user = repository.findById(id);
    if ("edit".equals(action)) {
        return ResponseEntity.ok(usersPageRenderer.renderEditForm(user));
    }
    return ResponseEntity.ok(usersPageRenderer.renderDetail(user));
}
```

**Cross-framework portability**: Every target framework supports this pattern:

| Framework | Mechanism | Multi-Format Declaration |
|-----------|-----------|------------------------|
| Spring Boot (Java) | `@GetMapping(produces = ...)` | Separate methods per `produces` value |
| FastAPI (Python) | `@app.get("/", response_class=...)` | Separate route functions or conditional |
| Gin (Go) | `c.NegotiateFormat(...)` | Switch on negotiated format |
| Sinatra (Ruby) | `respond_to` block | Format-specific blocks |
| Rocket (Rust) | `#[get("/", format = "...")]` | Separate handler per format attribute |

## Request/Response Flow

### Outbound (server → client): Response Body

```
Request arrives with Accept header
    │
    ▼
Framework matches Accept header against controller `produces` values
    │
    ├── Accept: application/json → JSON handler method
    │   → Controller returns domain object
    │   → Jackson serializes to JSON (existing behavior, no change)
    │
    ├── Accept: text/html → HTML handler method
    │   → Controller calls PageRenderer.renderList/Detail/EditForm()
    │   → PageRenderer builds content with escapeHtml() per field
    │   → PageRenderer loads template, replaces {{CONTENT}}, returns HTML string
    │   → Controller returns HTML string with Content-Type: text/html
    │
    └── No match → 406 Not Acceptable (framework handles this)
```

### Inbound (client → server): Form Submissions

```
Client submits HTML form
    │
    ├── Content-Type: application/json → JSON path (existing, no change)
    │
    └── Content-Type: application/x-www-form-urlencoded → form path
        → Framework parses form fields to Map<String, String>
        → Controller converts to domain object (type coercion)
        → Business logic, rule evaluation, save
        → Redirect (303 See Other) back to list or detail page (PRG pattern)
```

### HTML-Specific: Form Submission Handling

- **Hidden `_method` field**: HTML forms only support GET/POST. Edit forms use `method="POST"` with `<input type="hidden" name="_method" value="PUT">`. Spring's `HiddenHttpMethodFilter` converts to PUT before the controller sees it.
- **Checkbox fields**: Unchecked checkboxes are absent from form data. Treat missing boolean fields as `false`.
- **Type coercion**: Form data is all strings. Convert to correct types using model field metadata (baked into the PageRenderer or a shared utility at build time).
- **PRG pattern**: HTML form submissions redirect after success (POST → 303 See Other → GET). Prevents duplicate submissions on browser refresh.

## HTML Routing

### Same URLs, Accept Header Differentiation

| Request | Accept: application/json | Accept: text/html |
|---------|------------------------|-------------------|
| `GET /users` | JSON array | HTML list table |
| `GET /users/{id}` | JSON object | HTML detail view (read-only) |
| `GET /users/{id}?action=edit` | N/A (HTML only) | HTML edit form (pre-filled) |
| `GET /users?action=create` | N/A (HTML only) | HTML create form (static, empty) |
| `POST /users` | JSON response (201) | Redirect to list (303) |
| `PUT /users/{id}` | JSON response (200) | Redirect to detail (303) |
| `DELETE /users/{id}` | 204 No Content | Redirect to list (303) |
| `GET /views/user-profile` | JSON array | HTML list table (read-only, no action links) |

**Navigation links:**
- List page: "Create New" link → `GET /users?action=create`
- List page: per-row "View" link → `GET /users/{id}`
- Detail page: "Edit" link → `GET /users/{id}?action=edit`
- Edit form: "Cancel" link → `GET /users/{id}` (back to detail)
- Create form: "Cancel" link → `GET /users` (back to list)

### Default Format Behavior

| Scenario | Format Returned | Why |
|----------|----------------|-----|
| `Accept: application/json` | JSON | Explicit JSON request |
| `Accept: text/html` | HTML | Explicit HTML request (browser navigation) |
| `Accept: */*` | JSON | Wildcard — JSON listed first in `produces` |
| No `Accept` header | JSON | Missing header — JSON default |
| `Accept: text/plain` (unsupported) | 406 Not Acceptable | No matching handler |

## HTML Output Structure

Runtime HTML output must match the structure of `generated-html/` (Phase 0d static pages). Same page types, same field mappings, same navigation — but with real data.

### Field → HTML Input Mapping

Same mapping as `HtmlCrudGenerator` (Phase 0d):

| models.yaml type | original_sql_type | HTML Input | Notes |
|-----------------|-------------------|------------|-------|
| string | VARCHAR | `<input type="text">` | |
| string | TEXT | `<textarea>` | Check `original_sql_type` |
| int32/int64 | INT/BIGINT | `<input type="number" step="1">` | |
| decimal | DECIMAL | `<input type="number" step="0.01">` | |
| float64 | FLOAT/DOUBLE | `<input type="number" step="any">` | |
| bool | BOOLEAN | `<input type="checkbox">` | Never `required` |
| date | DATE | `<input type="date">` | |
| datetime | TIMESTAMP | `<input type="datetime-local">` | |

### Nullability

- `NOT NULL` column → `required` attribute on input
- Nullable column → no `required` attribute

If the model has rules in `specs.yaml`, form pages (create + edit) include:

## Testing Strategy

### Unit Tests (no server needed)

| Test | What It Verifies |
|------|-----------------|
| `HtmlEscapeUtils` escapes all 5 characters correctly | Security boundary |
| `HtmlEscapeUtils` handles null, empty, and Unicode input | Edge cases |
| `HtmlEscapeUtils` fuzz test — random strings never produce unescaped output | Exhaustive security |
| `*PageRenderer.renderList()` produces valid HTML table rows | List rendering |
| `*PageRenderer.renderDetail()` produces valid `<dl>` structure | Detail rendering |
| `*PageRenderer.renderEditForm()` pre-fills values with escaped data | Edit form rendering |
| `*PageRenderer.renderCreateForm()` returns static template (no substitution) | Create form rendering |
| Table output structure matches `generated-html/` list page (structural diff) | Consistency with Phase 0d |
| Form output structure matches `generated-html/` create/edit page (structural diff) | Consistency with Phase 0d |
| NOT NULL fields have `required` attribute in create/edit templates | Correct validation |
| Business rules rendered in form pages | Rule integration |

### Structural Diff Tests (unit level)

Compare runtime HTML output against `generated-html/` static pages:
1. Generate runtime HTML via `PageRenderer` with sample data
2. Strip data values (replace cell contents with placeholders)
3. Diff against `generated-html/` static pages (also stripped)
4. Structural elements must match: same tags, same attributes, same nesting

### HTTP Tests (server required, added to `tests/http-tests.yaml`)

| Test | Request | Expected |
|------|---------|----------|
| JSON default | `GET /users` (no Accept) | `Content-Type: application/json` |
| HTML via Accept | `GET /users` with `Accept: text/html` | `Content-Type: text/html`, contains `<table>` |
| HTML detail | `GET /users/{id}` with `Accept: text/html` | Contains `<dl>`, field values |
| HTML edit form | `GET /users/{id}?action=edit` with `Accept: text/html` | Contains `<form>`, pre-filled values |
| HTML create form | `GET /users?action=create` with `Accept: text/html` | Contains `<form>`, empty inputs |
| Form POST | `POST /users` with form data, `Accept: text/html` | 303 redirect to list |
| Form POST with blocking rule | `POST /users` with invalid data, `Accept: text/html` | Re-renders form with error messages |
| Unsupported format | `GET /users` with `Accept: text/plain` | 406 Not Acceptable |
| View HTML | `GET /views/user-profile` with `Accept: text/html` | Contains `<table>`, no edit/delete links |
| XSS prevention | `POST /users` with `<script>alert('xss')</script>` in field | Field value escaped in HTML response |

## MVP Scope

### In Scope (JSON + HTML only)

- Per-model PageRenderer classes (generated at build time)
- Static template files with `{{CONTENT}}` placeholders (pipeline-level artifact)
- `HtmlEscapeUtils` centralized security utility
- List pages with action links (View per row, Create New)
- Detail pages with Edit link
- Edit forms with pre-filled, escaped values (via `?action=edit`)
- Create forms (fully static, no data substitution)
- PRG pattern (Post/Redirect/Get) for HTML form submissions
- Re-render form with inline error messages on blocking rule violation (422)
- Navigation links (root index, list ↔ detail ↔ edit ↔ create)
- Business rules display on form pages
- Structural diff tests: runtime HTML vs `generated-html/` static pages
- XSS fuzz testing for `HtmlEscapeUtils`

### Out of Scope (future — same PageRenderer pattern)

- XML support (`Accept: application/xml`) — add `renderXml()` to PageRenderer
- CSV support (`Accept: text/csv`) — add `renderCsv()` to PageRenderer
- `?format=html` query param fallback
- CSS styling beyond minimal inline
- JavaScript of any kind
- Pagination, sorting, filtering in HTML views
- Authentication/session handling in HTML
- `Vary: Accept` header (best practice, but low priority for MVP — generated apps are not behind CDNs)

## Open-Source / Modularization Potential

The PageRenderer + `HtmlEscapeUtils` + template file pattern is framework-independent and could be extracted as a standalone module:

- **Template files**: Language-agnostic HTML layouts — reusable by any project that generates CRUD pages
- **`HtmlEscapeUtils`**: Per-language HTML escaping — small enough to be a standalone library or snippet
- **PageRenderer pattern**: Generated per-model renderers — the generation logic is appget-specific, but the output classes have zero appget dependencies

This aligns with the broader pipeline modularization goal (SQL parsing, protobuf generation, Gherkin parsing — all candidates for extraction).

## Decisions Log

All resolved during design review (2026-04-03):

| # | Question | Decision | Rationale |
|---|----------|----------|-----------|
| 1 | Detail vs Edit form rendering? | Detail (`<dl>`) by default. Edit form via `?action=edit` query parameter. | Cleaner separation — edit form HTML only rendered when explicitly requested. Smaller response payload, smaller attack surface. |
| 2 | Model metadata in transform? | Build-time generated. PageRenderer has all field metadata baked in as static data. No runtime YAML loading. | Matches existing patterns (SpecificationRegistry). Aligns with build-time principle. |
| 3 | Create form access? | "Create New" link on list page → `?action=create`. "Edit" link on detail page → `?action=edit`. | Simple navigation. Uses existing POST/PUT endpoints for form submissions. |
| 4 | Single converter vs per-model? | Per-model PageRenderer classes. No global converter. | Eliminates runtime model identification problem entirely. Each renderer knows its model at build time. |
| 5 | Framework-native templating? | **No.** Appget-owned templates + PageRenderers. No Thymeleaf, Jinja2, ERB. | Eliminates version coupling, CVE exposure, and framework lock-in. LLM-auditable, fuzz-testable. |
| 6 | Template complexity? | Minimal — `{{CONTENT}}` placeholder only. Templates are dumb static layouts. | All logic in PageRenderer (generated code). Templates are human-readable HTML files. |
| 7 | Where does escaping live? | Centralized `HtmlEscapeUtils` utility class per language. | Single security boundary. Auditable in seconds. Fuzz-testable. Same 5-char replacement in every language. |

## Research Reference

See [RESEARCH-content-negotiation-survey.md](RESEARCH-content-negotiation-survey.md) for the full content negotiation survey (completed 2026-04-02/03). Key findings that informed this design:

1. No existing tool generates HTML transforms — appget's PageRenderer fills this gap
2. Every target framework has built-in Accept header parsing — appget leverages this, does not reimplement
3. Framework templating engines (Thymeleaf, Jinja2) are not required — appget generates its own rendering
4. The `produces` annotation pattern is portable across all target frameworks
