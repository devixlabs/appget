# SPEC: HTML CRUD Code Generation — Java Reference Implementation

## Context & Priority

This is the next major feature for the Java reference implementation and a **root-level
pipeline concern** — HTML output sits alongside `openapi.yaml` as a generic artifact
produced from the pipeline, not inside a language subproject. It will be reusable across
all future language subprojects with minimal adaptation.

The Java subproject is the first and only complete implementation. HTML codegen is
designed here first, then extracted to the root pipeline once the pattern is proven.

---

## Goal

Generate minimal, functional static HTML CRUD pages directly from the pipeline's existing
intermediate representations (`models.yaml`, `specs.yaml`). No JavaScript frameworks, no
CSS dependencies, no build tooling. Pure HTML with native browser form semantics.

The generated HTML is a **first-class pipeline artifact** — same standing as `openapi.yaml`.

---

## Design Constraints

- **HTML elements only**: `<form>`, `<input>`, `<select>`, `<textarea>`, `<table>`,
  `<a href>`, `<button type="submit">`, `<label>`. Nothing beyond standard HTML5.
- **No JavaScript** — not even inline scripts. If the browser can't do it natively, skip it.
- **No external CSS** — inline `style` attributes are acceptable for minimal readability,
  but not required. Functional > pretty.
- **No JavaScript frameworks, no npm, no build step.**
- Form actions must align exactly with REST routes generated by `AppServerGenerator`.
- Output is git-ignored (generated artifact, like all other generated outputs).
- Generator uses **StringBuilder** — consistent with `AppServerGenerator`,
  `ProtoOpenAPIGenerator`, `ModelsToProtoConverter`. Do NOT introduce a templating library.

---

## Pipeline Position

```
models.yaml + specs.yaml
        │
        ▼
HtmlCrudGenerator
        │
        ▼
generated-html/
  ├── index.html              ← root nav: all domains + entities
  ├── {domain}/
  │   └── {table}/
  │       ├── index.html      ← list/table view (GET all)
  │       ├── create.html     ← new record form (POST)
  │       ├── edit.html       ← edit form (PUT/PATCH) — uses query param ?id=
  │       └── view.html       ← read-only detail (GET by id)
```

This mirrors how `openapi.yaml` is generated from `.proto` files — a single generator
consuming pipeline intermediates, producing a complete artifact set.

---

## Inputs

### Primary: `models.yaml`
The main data source. Provides:
- Table names (entity names)
- Column names, types, nullability
- Domain assignments
- View definitions (separate handling — see below)

### Secondary: `specs.yaml`
Optional enrichment. Provides:
- Business rule targets and rule names
- Blocking vs informational rule classification
- Status outcome labels

The generator must function correctly with or without `specs.yaml` (graceful degradation).

---

## Output Pages Per Entity (Base Table)

### 1. `index.html` — List View
- `<table>` with one column per model field (use column names as headers)
- One row per record (data is placeholder/static — generator emits structure only)
- Action column: `<a href="view.html?id=">View</a>` | `<a href="edit.html?id=">Edit</a>`
  | `<form method="POST" action="/api/{domain}/{table}/{id}">` with DELETE override or
  a dedicated delete endpoint per `AppServerGenerator` routes
- `<a href="create.html">New {Entity}</a>` link at top

### 2. `create.html` — Create Form
- `<form method="POST" action="/api/{domain}/{table}">`
- One `<input>` per non-PK, non-auto field (see field type mapping below)
- `NOT NULL` columns → `required` attribute on input
- Primary key field omitted (auto-assigned server-side)
- `<button type="submit">Create</button>`
- `<a href="index.html">Cancel</a>`

### 3. `edit.html` — Edit Form
- Same structure as `create.html`
- `<form method="POST" action="/api/{domain}/{table}/{id}">` (REST PUT via hidden input
  or direct POST — align with what `AppServerGenerator` emits for updates)
- Include hidden `<input type="hidden" name="_method" value="PUT">` if needed
- PK field rendered as `<input type="hidden">` (not editable)
- `<button type="submit">Save</button>`
- `<a href="index.html">Cancel</a>`

### 4. `view.html` — Read-Only Detail
- `<table>` or `<dl>` layout: field name | value (placeholder)
- No form elements
- `<a href="edit.html?id=">Edit</a>` | `<a href="index.html">Back</a>`

## Field → HTML Input Type Mapping

| models.yaml type   | HTML element                                        | Notes                          |
|--------------------|-----------------------------------------------------|--------------------------------|
| VARCHAR, CHAR      | `<input type="text">`                               |                                |
| TEXT               | `<textarea>`                                        | Long-form content              |
| INT, INTEGER       | `<input type="number" step="1">`                    |                                |
| BIGINT             | `<input type="number" step="1">`                    |                                |
| DECIMAL, NUMERIC   | `<input type="number" step="0.01">`                 |                                |
| FLOAT, DOUBLE      | `<input type="number" step="any">`                  |                                |
| BOOLEAN            | `<input type="checkbox">`                           |                                |
| DATE               | `<input type="date">`                               |                                |
| TIMESTAMP,DATETIME | `<input type="datetime-local">`                     |                                |
| ENUM (if present)  | `<select>` with `<option>` per value                | Parse enum values from schema  |

Every input must have a `<label>` with `for` attribute matching input `id`.
Use column name (snake_case) as the `name` attribute — matches REST payload expectations.

---

## Nullability → HTML Validation

- `NOT NULL` column → add `required` attribute to the input element
- Nullable column → no `required` attribute

This is the only client-side validation. No custom validation logic.

---

## Business Rules Integration (from `specs.yaml`)

If `specs.yaml` is available and a rule targets this entity:
- Render a `<details><summary>Business Rules</summary>...</details>` block on the
  `create.html` and `edit.html` pages
- List rule name, target field(s), operator, value, and outcome labels as plain text
- Mark `@blocking` rules visually (e.g., `[BLOCKING]` text prefix)
- This is informational only — no client-side enforcement

This gives human developers immediate visibility into what the server will validate,
without any JS required.

---

## SQL Views → HTML

Views from `views.sql` get a read-only list page only:
- `view-index.html` — `<table>` with view columns as headers
- No create/edit/delete (views are read-only by definition)
- Link from root `index.html` under the domain section

---

## Root `index.html` — Navigation Hub

Top-level generated file at `generated-html/index.html`:
- One section per domain
- Per domain: list of entity links to their `index.html`
- Per domain: list of view links to their `view-index.html`
- Title: "AppGet — Generated CRUD"

This is the entry point for anyone opening the generated HTML output.

## REST Route Alignment with `AppServerGenerator`

The HTML form actions must match the routes `AppServerGenerator` emits for Spring Boot.
The Agent must read `AppServerGenerator` source to confirm the exact route pattern before
hardcoding anything. Expected convention (verify against source):

```
GET    /api/{domain}/{table}          → index.html list
GET    /api/{domain}/{table}/{id}     → view.html detail
POST   /api/{domain}/{table}          → create.html form action
PUT    /api/{domain}/{table}/{id}     → edit.html form action
DELETE /api/{domain}/{table}/{id}     → delete form action in index.html
```

If `AppServerGenerator` uses a different convention, HTML must follow it — not the other way around.

## New Generator Class

**`HtmlCrudGenerator.java`** in `src/main/java/` alongside existing generators.

- Follows the same constructor/entry-point pattern as `AppServerGenerator` and
  `ProtoOpenAPIGenerator` — read those first as implementation reference
- Consumes parsed `models.yaml` (already parsed by `SQLSchemaParser` in pipeline)
- Optionally consumes parsed `specs.yaml`
- Uses `CodeGenUtils.java` for any string transforms needed
- Outputs to configurable directory (default: `src/main/resources/static/generated-html/`
  or a standalone `generated-html/` at project root — Agent decides based on Spring Boot
  static resource conventions)

```

```

include HTML going forward.

Add `generated-html/` to `.gitignore` — same treatment as `src/main/java-generated/`
and `models.yaml`.

## Out of Scope for v1

- CSS styling beyond minimal inline style (if any)
- JavaScript of any kind
- Authentication / session handling in HTML
- Pagination controls (list page shows all rows structurally)
- File upload inputs
- Relational dropdowns (FK fields render as plain `<input type="number">` for the ID)
- Real data binding — generated HTML is structural, not data-connected

These are deliberate exclusions, not oversights. The goal is a provably correct
structural scaffold, not a production UI.
